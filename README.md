# IT-SP Task Planner

Веб-приложение для регистрации, обработки и контроля заявок ИТ-поддержки с ролевым доступом и синхронизацией данных через SeaTable.

## Что это за проект

`IT-SP Task Planner` — это единая система заявок, где:

- сотрудники создают заявки;
- руководитель контролирует выполнение;
- администратор управляет пользователями, направлениями и общими настройками.

Проект сделан как статический фронтенд + serverless API, а источником данных выступает SeaTable.

## Основные возможности

- создание и редактирование заявок;
- статусы и история изменений;
- фильтрация, поиск, приоритеты и SLA;
- управление направлениями техподдержки;
- управление пользователями и ролями;
- вложения в заявках;
- отчеты и сводная статистика;
- синхронизация с SeaTable.

## Роли в системе

- `admin` — полный доступ (пользователи, направления, все заявки, настройки);
- `director` — расширенный доступ к заявкам по своей зоне ответственности;
- `employee` — создание заявок и работа со своими заявками в рамках ограничений.

## Технологии

- Frontend: Vanilla JavaScript, HTML, CSS
- Backend: Node.js serverless-функции (папка `api/`)
- Хранилище данных: SeaTable API
- Хостинг/деплой: Vercel
- DNS/edge: Cloudflare

## Структура проекта

- `index.html` — основной UI
- `js/app.js` — клиентская бизнес-логика (авторизация, UI, запросы к API)
- `css/` — стили
- `api/_seatable.js` — общий адаптер для работы с SeaTable
- `api/tasks/` — CRUD для заявок
- `api/users/` — CRUD для пользователей
- `api/directions/` — CRUD для направлений
- `api/health.js` — healthcheck для проверки доступности SeaTable
- `test/seatable.test.js` — автотесты основных сценариев API
- `vercel.json` — настройки маршрутизации/кэширования для Vercel
- `_redirects` — дополнительное правило маршрутизации API

## Как работает синхронизация с SeaTable

1. Фронтенд отправляет запросы на относительные API-эндпоинты (`/api/...`).
2. Serverless-функции валидируют и преобразуют данные.
3. Запросы уходят в SeaTable через app access token.
4. После операций записи выполняется проверка (re-fetch), чтобы подтвердить, что изменения реально сохранены.
5. UI получает каноничные данные и обновляется.

Важно: SeaTable является источником истины для задач, направлений и пользователей.

## Переменные окружения

Минимально необходимые переменные:

- `SEATABLE_API_TOKEN`
- `SEATABLE_BASE_UUID`

Опционально:

- `SEATABLE_SERVER` (по умолчанию `https://cloud.seatable.io`)
- `SEATABLE_TABLE_NAME`
- `SEATABLE_VIEW_NAME`
- `SEATABLE_USERS_TABLE`
- `SEATABLE_DIRECTIONS_TABLE`

Рекомендуется хранить локально в `.env.local` и в Vercel Environment Variables для production.

## Деплой (кратко)

1. Проверить переменные окружения в Vercel.
2. Убедиться, что DNS домена настроен корректно.
3. Задеплоить актуальную ветку в production.
4. Проверить:
   - главную страницу,
   - `/api/health`,
   - создание/обновление заявок,
   - синхронизацию пользователей/направлений.

## Диагностика типовых проблем

### 1) Сайт открывается, но API отвечает 403

Проверить WAF/Challenge правила Cloudflare для `/api/*` и отключить блокирующие действия для API.

### 2) Новая задача видна сразу, но пропадает после перезагрузки

Это означает, что запись не подтверждена в SeaTable. Проверять серверные логи `api/tasks` и результат верификации после `POST`.

### 3) Вложение не появляется в SeaTable

- проверить тип колонки `attachments` в SeaTable (должен быть `File`);
- проверить права токена на upload файлов;
- смотреть логи по `attachment upload failed`.

### 4) Ошибка вида `id: '$id'` или `numericId: NaN`

Проблема в роутинге/rewrites. Использовать корректный route handler `api/tasks/[id].js` без подстановки литерала `$id`.

## Для отчета (краткая формулировка)

Проект представляет собой систему Service Desk для ИТ-поддержки с ролевой моделью доступа, построенную на JavaScript-фронтенде и serverless API на Node.js. Данные централизованно хранятся в SeaTable, синхронизация выполняется через REST API с верификацией операций записи. Развертывание осуществляется в Vercel, сетевой слой и DNS — через Cloudflare.
