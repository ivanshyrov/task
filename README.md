# Система Service Desk «Планировщик задач»

Веб-приложение для регистрации, обработки и контроля заявок технической поддержки. Данные хранятся в **SeaTable**, backend — **serverless API на Vercel** (Node.js), интерфейс — **Vanilla JS + HTML + CSS**.

## 1. Назначение системы

Корпоративная система Service Desk для автоматизации процессов технической поддержки.

**Возможности:**

- сотрудникам — создавать заявки и отслеживать свои обращения;
- администраторам — вести все заявки, назначать исполнителей, управлять справочниками и пользователями;
- контроль сроков (SLA), отчёты и фильтрация;
- уведомления о событиях с заявками;
- синхронизация данных с SeaTable через API.

## 2. Ролевая модель

| Роль | Описание | Основные возможности |
|------|----------|----------------------|
| **Администратор** | Полный доступ | Все заявки, пользователи, направления, базы, отчёты, массовое удаление |
| **Сотрудник СП** | Ограниченный доступ | Создание заявок, просмотр и редактирование **своих** заявок по правилам UI и сервера |

### Учётные записи по умолчанию (при первой инициализации в коде)

При отсутствии пользователя `admin` в таблице Users приложение может создать запись с паролем из дефолта (см. `js/app.js`, `DEFAULT_USERS`). В продакшене рекомендуется задать пароли в SeaTable вручную и хранить **SHA-256** хэш в колонке `password_hash`.

| Логин | Пароль (дефолт в коде) | Роль |
|-------|------------------------|------|
| admin | admin123 | Администратор |
| employee | employee123 | Сотрудник СП |

### Особенности пользователей

- У администратора в одном поле ФИО можно перечислить несколько имён через запятую — логика UI трактует их как несколько «личностей» для назначения и отображения.
- **Уникальность логина**: в SeaTable не должно быть нескольких строк с одним `username` — иначе при логине может подставляться «первая попавшаяся» строка.

## 3. Заявки, SLA, отчёты

### Статусы и переходы

```
Новая → В работе → Завершена
    ↓              ↓
Отклонена      (финал)
```

- **Администратор** — переводы по правилам переходов.
- **Сотрудник** — для своих заявок: из «Новой» в «В работе», из «В работе» в «Завершена» (см. `js/app.js`, `STATUS_TRANSITIONS`).

### Приоритеты и SLA (дней)

| Приоритет | SLA |
|-----------|-----|
| Критический | 1 |
| Высокий | 2 |
| Средний | 3 |
| Низкий | 5 |

### Отчёты

Фильтры по базе, исполнителю (админ), периоду; простая и детальная статистика. Сотрудник видит только свои задачи в отчётах.

Подробные сценарии использования (создание заявки, фильтры, настройки темы и т.д.) сохранены в логике `index.html` и `js/app.js`.

## 4. Уведомления

Система уведомлений информирует пользователей о событиях с заявками. Уведомления отображаются в интерфейсе и хранятся локально.

### Хранение

- Уведомления сохраняются в **localStorage** браузера.
- Ключ записи: `taskPlannerNotifications_v2_<username>` (уникален для каждого пользователя).
- Максимум **20 записей** на пользователя. При превышении старые уведомления удаляются.

### Правила формирования

**Сотрудник (role === 'employee')** получает личные уведомления только если он автор заявки:

| Событие | Сообщение |
|---------|-----------|
| Создана | «Ваша заявка #… создана…» |
| Выполнена | «Ваша заявка #… выполнена…» |
| Отклонена | «Ваша заявка #… отклонена…» |

**Администраторы** получают уведомления о всех заявках. Если сотрудник создаёт заявку — уведомление приходит первому админу из списка пользователей. Администратор-автор не получает дублирующее уведомление «Ваша заявка» — он уже видит информацию в общей ленте.

### Интерфейс

- При входе пользователя (`initApp()`) вызывается `loadUserNotifications()` — загружается список уведомлений текущего пользователя.
- При выходе (`logoutUser()`) счётчик непрочитанных уведомлений сбрасывается.
- При открытии модалки уведомлений все текущие уведомления помечаются как прочитанные и сохраняются через `persistCurrentUserNotifications()`.
- Счётчик непрочитанных уведомлений отображается в UI (красный бейдж).

### Сопоставление автора

Функция `resolveUsernameByFullName(fullName)` ищет пользователя в таблице Users:

1. Точное совпадение по полю `full_name`.
2. Поиск по частям через запятую (формат «Фамилия, Имя Отчество»).

Это позволяет корректно определять автора заявки даже при разных форматах записи ФИО.

### Реализация

Логика уведомлений реализована в `js/app.js`:

- Функции: `loadUserNotifications()`, `persistCurrentUserNotifications()`, `resolveUsernameByFullName()`, `createNotificationRecord()`.
- Интеграция: `initApp()`, `logoutUser()`, `createTask()`, `updateTaskFromModal()`.
- UI: секция отображения уведомлений и счётчик в заголовке.

## 6. Аутентификация и безопасность

### Серверная сессия

- Вход: `POST /api/auth/login` — проверка логина/пароля по таблице Users, выдача **подписанной cookie** `tp_session` (HttpOnly, SameSite=Lax, Secure в production).
- Текущий пользователь: `GET /api/auth/me`.
- Выход: `POST /api/auth/logout`.

Сессия **stateless**: в cookie хранится только подписанный payload с ограниченным набором полей (**без** больших данных вроде base64-аватара), чтобы не превысить лимит размера cookie (~4 KB).

### Пароли

- В таблице Users поле `password_hash` — **SHA-256** от пароля в шестнадцатеричном виде (как считает клиентский `hashPassword` и серверный `sha256` в `api/_auth.js`).
- Хэш пароля **не возвращается** в ответах `GET /api/users`.

### Авторизация API

- Защищённые маршруты проверяют сессию и роль (`ensureAuth` в `api/_auth.js`).
- Задачи: сотрудник на сервере получает только заявки, где `author` совпадает с **ФИО** из сессии; администратор — все.
- Справочники directions/activity: изменение только у роли **admin**; чтение — у admin и employee.

### CORS и заголовки

- Разрешённые origin задаются переменной **`ALLOWED_ORIGINS`** (см. ниже). Запросы с credentials используют точное совпадение `Origin`.
- В `vercel.json` заданы заголовки безопасности и CSP (без `unsafe-eval` для скриптов проекта).

### Rate limiting

Часть эндпоинтов ограничивает число запросов с одного IP в минуту (см. `api/tasks/index.js`, `api/directions/index.js`, и т.д.).

### Клиент

- Сессия в браузере: `fetch` с `credentials: 'include'`.
- Таймаут бездействия ~30 минут — выход из приложения (см. `SESSION_TIMEOUT` в `js/app.js`).

## 7. Переменные окружения

### Обязательные для SeaTable

| Переменная | Описание |
|------------|----------|
| `SEATABLE_API_TOKEN` | API-токен приложения SeaTable |
| `SEATABLE_BASE_UUID` | UUID базы (dtable) |

### Опционально SeaTable

| Переменная | По умолчанию / назначение |
|------------|---------------------------|
| `SEATABLE_SERVER` | `https://cloud.seatable.io` или URL self-hosted |
| `SEATABLE_TABLE_NAME` | `Tasks` |
| `SEATABLE_VIEW_NAME` | `Default` |
| `SEATABLE_USERS_TABLE` | `Users` |
| `SEATABLE_DIRECTIONS_TABLE` | `Directions` |
| `SEATABLE_ACTIVITY_TABLE` | `ActivityDirections` или имя вашей таблицы |

### Обязательные для продакшена (безопасность и CORS)

| Переменная | Описание |
|------------|----------|
| `SESSION_SECRET` | Длинная случайная строка для подписи cookie-сессии (обязательно задать в Vercel Production). |
| `ALLOWED_ORIGINS` | Список разрешённых origin через запятую, без пробелов. Пример: `https://your-domain.ru,https://www.your-domain.ru` |

Локально можно использовать `.env.local` (файл в `.gitignore`). На Vercel переменные задаются в **Project → Settings → Environment Variables**.

## 9. SeaTable: таблицы и колонки

Имена таблиц можно переопределить переменными окружения (см. раздел 7).

### Tasks (задачи)

Основные поля: `id`, `title`, `description`, `author`, `assignee`, `department`, `priority`, `status`, `deadline`, `report`, `rejected_reason`, `created_at`, `updated_at`, SLA-поля, комментарии/история/вложения (JSON в ячейках по логике `_seatable.js`).

### Users

| Колонка | Назначение |
|---------|------------|
| `username` | Логин (уникальный) |
| `full_name` | ФИО |
| `role` | `admin` / `employee` |
| `department`, `position`, `email`, `phone`, `office` | Профиль |
| `password_hash` | SHA-256 пароля (hex), не отдаётся клиенту в API |
| `avatar` | Опционально; большие base64 в cookie не кладутся |

### Directions / Activity

Таблицы-справочники с колонкой `name` (и `_id` для строк).

## 10. HTTP API

Все перечисленные ниже маршруты (кроме входа) требуют **валидной cookie-сессии** после логина.

### Аутентификация

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/api/auth/login` | Body: `{ "username", "password" }` → установка cookie |
| GET | `/api/auth/me` | Текущий пользователь |
| POST | `/api/auth/logout` | Сброс сессии |

### Задачи

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/tasks` | Список задач (для сотрудника — только свои по автору) |
| POST | `/api/tasks` | Создание |
| GET | `/api/tasks/[id]` | Одна задача |
| PUT | `/api/tasks/[id]` | Обновление |
| DELETE | `/api/tasks/[id]` | Удаление |

Дополнительно в `api/tasks/index.js` поддерживаются PUT/DELETE телом запроса для массовых операций там, где это предусмотрено фронтом.

### Пользователи

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/users` | Список (без `password_hash`) |
| POST | `/api/users` | Создание (admin) |
| PUT | `/api/users?username=` | Обновление (admin) |
| DELETE | `/api/users?username=` | Удаление (admin) |

### Справочники

| Метод | Путь | Описание |
|-------|------|----------|
| GET/POST/PUT/DELETE | `/api/directions` | Направления техподдержки |
| GET/POST/PUT/DELETE | `/api/activity` | Направления деятельности |

### Health

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/health` | Проверка связи с SeaTable (требует сессию) |

## 11. Локальная разработка и тесты

### Зависимости

```bash
npm install
```

### Тесты

```bash
npm run test
```

Запускает `test/seatable.test.js` (маппинг SeaTable, форматы update/delete, смоки хендлеров).

### Локальный запуск API

Точка входа в `package.json` указывает на пример запуска одного файла; для полного UI обычно используют статический сервер или `vercel dev`.

Пример с переменными из `.env.local`:

```bash
# после настройки .env.local
npx vercel dev
```

Откройте локальный URL, который выдаст Vercel CLI.

## 12. Деплой на Vercel

1. Подключить репозиторий к Vercel или использовать CLI: `vercel`, затем `vercel --prod`.
2. В **Environment Variables** добавить все переменные из раздела [7](#7-переменные-окружения), для Production (и при необходимости Preview).
3. Убедиться, что **`ALLOWED_ORIGINS`** содержит точный HTTPS-origin вашего сайта (без слэша в конце).
4. Задать **`SESSION_SECRET`** (длинная случайная строка).
5. Выполнить деплой и проверить вход, затем загрузку `/api/tasks` и справочников без 401.

Скрипты из `package.json`: `deploy:preview`, `deploy:prod` (требуют `VERCEL_TOKEN` в CI).

## 13. Что можно улучшить

**Функционально:** email/SMS уведомления, расширенная история, шаблоны заявок, экспорт отчётов, календарь.

**Технически:** Argon2id/bcrypt вместо SHA-256 для паролей с миграцией при входе; серверный rate-limit на `/api/auth/login`; Redis для лимитов в мульти-инстанс окружении; строгая Zod-валидация тел запросов.

**Интеграции:** webhooks, Telegram/Slack — по необходимости.
