# IT-SP · Планировщик задач + SeaTable — обзор кодовой базы и диагностика текущих проблем

## Summary
Это одностраничный веб‑планировщик задач на чистом HTML/CSS/JavaScript с serverless‑бэкендом для SeaTable. Пользовательский интерфейс живёт в браузере, а CRUD по задачам и пользователям идёт через API‑роуты на стороне хостинга, чтобы токен SeaTable не попадал на клиент.

По сути это не «обычный сайт», а связка из трёх частей: браузерный UI, серверные функции и внешняя база данных SeaTable. Главная рабочая модель — локальный UI‑состояние + синхронизация с SeaTable, поэтому проект чувствителен и к фронтенд‑ошибкам, и к инфраструктуре домена, и к корректности переменных окружения.

## Architecture
### Архитектурный стиль
Архитектура гибридная:

- **SPA без фреймворка**: вся логика интерфейса сосредоточена в `js/app.js`.
- **Serverless backend**: обработчики в `api/` рассчитаны на Vercel.
- **Адаптер SeaTable**: `api/_seatable.js` прячет различия между SeaTable v1/v2, форматами ответов и способом авторизации.
- **Локальное состояние + удалённая синхронизация**: данные частично живут в `localStorage`, частично в SeaTable, а UI оптимистично обновляется до подтверждения API.

### Как запускается приложение
1. Браузер открывает `index.html`.
2. Подключается `css/style.css` и `js/app.js`.
3. Скрипт сразу инициализирует пользователей из `localStorage` и, если возможно, подтягивает их из `/api/users`.
4. После логина приложение отображает основной интерфейс и синхронизирует задачи через `/api/tasks`.
5. На серверной стороне `/api/tasks` и `/api/users` ходят в SeaTable через `api/_seatable.js`.

### Технологический стек
- HTML5 + CSS3
- Vanilla JavaScript
- `fetch`, `AbortController`, `localStorage`, `crypto.subtle`
- Node.js serverless functions
- `node-fetch` как зависимость для серверной части
- SeaTable Cloud / SeaTable API
- Vercel deployment model, с отдельным legacy‑файлом для Netlify

## Key Abstractions

### `js/app.js` — главный клиентский контроллер
- **Responsibility**: управляет всеми экранами, формами, фильтрами, таблицами, модалками, профилем, уведомлениями и локальным состоянием.
- **Interface**: `bootstrapSession()`, `initApp()`, `applyRole()`, `switchView()`, `renderTasks()`, `createTask()`, `updateTaskFromModal()`, `editUser()`, `deleteUser()`, `loadSettings()`.
- **Lifecycle**: запускается сразу после загрузки страницы, остаётся активным всё время жизни вкладки.
- **Used by**: фактически всем фронтендом; это единственная точка, где UI собирается в рабочее состояние.

### `api/_seatable.js` — общий SeaTable-адаптер
- **Responsibility**: авторизация в SeaTable, выбор правильной базы API, повторные попытки, таймауты, нормализация форматов данных.
- **Interface**:
  - `getAppAccessToken()`
  - `getRowsBaseUrl(accessMeta)`
  - `seatableRequest(accessToken, url, options)`
  - `mapRowToTask(row)`
  - `mapTaskToRow(task)`
  - `buildUpdateRequestBody(...)`
  - `buildDeleteRequestBody(...)`
- **Lifecycle**: загружается серверными роутами при каждом запросе; держит in-memory cache токена между вызовами в рамках warm instance.
- **Used by**: `api/tasks/index.js`, `api/tasks/[id].js`, `api/users/index.js`.

### `api/tasks/index.js` — массовые операции по задачам
- **Responsibility**: list/create/update/delete задач в SeaTable.
- **Interface**: HTTP методы `GET`, `POST`, `PUT`, `DELETE`.
- **Lifecycle**: вызывается браузером через `/api/tasks`.
- **Used by**: `js/app.js` для синхронизации списка, создания, обновления и удаления задач.

### `api/tasks/[id].js` — точечный CRUD по задаче
- **Responsibility**: работа с одной задачей по числовому `id`, включая fallback на поиск `row_id`.
- **Interface**: HTTP методы `GET`, `PUT`, `DELETE`.
- **Lifecycle**: используется там, где нужно гарантированно адресовать конкретную запись.
- **Used by**: часть сценариев обновления/удаления из фронтенда и проверки консистентности.

### `api/users/index.js` — управление пользователями
- **Responsibility**: list/create/update/delete пользователей в SeaTable.
- **Interface**: HTTP методы `GET`, `POST`, `PUT`, `DELETE`.
- **Lifecycle**: вызывается из `js/app.js` при инициализации и админских операциях.
- **Used by**: управление аккаунтами, профилем и локальным кэшем пользователей.

### `SeaTableAdapter` в `js/app.js`
- **Responsibility**: конвертирует фронтенд-модель задачи в формат API и обратно.
- **Interface**: `toRecord(task)` / `fromRecord(record)`.
- **Lifecycle**: живёт внутри фронтенда как локальный слой сериализации.
- **Used by**: локальные представления, синхронизация, таблицы и карточки задач.

### `vercel.json`
- **Responsibility**: SPA rewrite на `index.html`, заголовки кеширования и CORS для API.
- **Interface**: `rewrites`, `headers`.
- **Lifecycle**: применяется на этапе деплоя.
- **Used by**: Vercel runtime и маршрутизация домена.

## Data Flow

### 1) Авторизация и старт приложения
1. `index.html` загружает `js/app.js`.
2. `bootstrapSession()` читает `localStorage` и вызывает `initUsers()`.
3. `initUsers()` сначала поднимает локальных пользователей, затем пытается слить их с `/api/users`.
4. Если сессия есть, `initApp()` грузит задачи, настройки, справочники и интерфейс.

### 2) Загрузка задач
1. `initApp()` вызывает `syncTasksFromApi()`.
2. Фронтенд делает `GET /api/tasks`.
3. `api/tasks/index.js` ходит в SeaTable, получает строки и нормализует их через `mapRowToTask()`.
4. Возвращённые задачи попадают в локальные `databases[]`.
5. `renderTasks()` перерисовывает таблицу и мобильные карточки.

### 3) Создание/обновление/удаление задачи
1. Пользователь открывает модалку.
2. Фронтенд собирает данные и валидирует форму.
3. Локально задача сразу добавляется/изменяется/удаляется.
4. Потом отправляется `POST`/`PUT`/`DELETE` на `/api/tasks` или `/api/tasks/:id`.
5. Если SeaTable возвращает ошибку, локальное состояние откатывается.

### 4) Управление пользователями
1. Админ открывает форму пользователя.
2. `addUser()` / `editUser()` / `deleteUser()` обновляют `localStorage`.
3. Далее отправляется запрос в `/api/users`.
4. На старте приложения `initUsers()` пытается синхронизировать локальные учетные записи с SeaTable без паролей.

### 5) Профиль и аватар
1. Клик по `#profileBtn` открывает `#profileModal`.
2. Профиль редактируется локально и сохраняется через `editUser()`.
3. Аватар хранится в `localStorage` как data URL с ключом `avatar_<username>`.

## Non-Obvious Behaviors & Design Decisions

### 1) Почему меню, профиль и бургер «пропали»
По коду эти элементы **не удалены** — они действительно есть в `index.html`:

- бургер: `.burger-btn` с `#burgerBtn`
- профиль: `#profileBtn`
- роль/выход: `#roleDisplay`

Если визуально они исчезли, наиболее вероятные причины такие:

1. **Не загрузился Font Awesome CDN**  
   Почти все эти элементы рисуются только иконками:
   - бургер `fa-bars`
   - профиль `fa-user-circle`
   - роль `fa-user-shield`
   - уведомления `fa-bell`

   Если `cdnjs.cloudflare.com` заблокирован, недоступен, или SRI/сеть не пропускают ресурс, кнопки становятся почти невидимыми.

2. **JS не завершил инициализацию**  
   Если `app.js` падает до `bootstrapSession()` / `initApp()`, часть интерфейса останется скрытой или не получит обработчики.

3. **Пользователь не вошёл в систему**  
   Пока `#app` скрыт, header/sidebar не отображаются вообще. Тогда кажется, что «меню пропало», хотя на самом деле интерфейс не перешёл в рабочий режим.

4. **На мобильном меню по умолчанию спрятано**  
   Sidebar стартует со `left:-100%` и появляется только через бургер. Если иконка бургера не видна, доступ к меню практически теряется.

### 2) Почему сайт по `taskspyanao.ru` может не открываться
С высокой вероятностью это **инфраструктурная проблема, а не чисто кодовая**. В репозитории есть несколько признаков этого:

- проект ориентирован на **Vercel** (`vercel.json`, `api/`)
- одновременно присутствует **Netlify function** (`netlify/functions/seatable-api.js`)
- `package.json` не поднимает HTTP‑сервер, а запускает `node api/tasks/index.js`, что **не является полноценным веб‑сервером**
- `index.html` принудительно редиректит на HTTPS для всех не‑локальных хостов

Из этого следует:

1. **Если домен не привязан к Vercel-проекту**, сайт будет недоступен.
2. **Если DNS/SSL не настроены**, HTTPS‑редирект в `index.html` сразу проявит проблему.
3. **Если домен ведёт на host, который ожидает `npm start` как сервер**, этот проект там не поднимется, потому что он serverless/SPA, а не классический Node‑сервер.
4. **Если домен открыт на другом провайдере**, Vercel‑rewrites и `/api/*` могут не работать вообще.

### 3) Смешение Vercel и Netlify — операционный риск
В одном репозитории живут:
- Vercel конфиг и Vercel API routes
- Netlify function как альтернативная реализация

Это опасно, потому что легко получить ситуацию, когда:
- фронтенд развернут на одной платформе,
- backend функции — на другой,
- домен смотрит на третью конфигурацию,
- а фактические `/api/tasks` и `/api/users` не существуют на том хосте, куда смотрит DNS.

### 4) Авторизация на клиенте не является безопасной
Сейчас безопасность системы фактически строится на:
- скрытом UI,
- localStorage,
- SHA-256 хешировании пароля в браузере,
- отсутствии server-side auth на `/api/tasks` и `/api/users`.

Это значит:
- любой, кто достучится до API, может вызывать CRUD,
- клиентская «сессия» не защищает backend,
- SHA-256 без соли хранится в браузере и не даёт настоящей защиты от офлайн‑атаки,
- UI‑права — это не security boundary.

### 5) CORS открыт слишком широко
`vercel.json` задаёт:

- `Access-Control-Allow-Origin: *`
- разрешены `GET, POST, PUT, DELETE, OPTIONS`

Это удобно для отладки, но небезопасно для публичного API. Если endpoint доступен извне, его можно дергать с любого origin.

### 6) SeaTable используется как source of truth, но с verification‑логикой
Роуты задач и пользователей после обновления часто:
- перечитывают запись,
- сравнивают результат,
- и при несовпадении могут вернуть 500.

Это сделано специально, чтобы поймать eventual consistency SeaTable, но на практике может выглядеть как «обновление прошло, но API ругнулся».

### 7) Задачи и пользователи частично кэшируются локально
Фронтенд сохраняет:
- задачи,
- сотрудников,
- направления,
- настройки,
- аватары,
- активную сессию.

Это ускоряет работу, но означает, что локальные данные могут расходиться с SeaTable, если синхронизация упала.

### 8) Требование HTTPS критично
В `index.html` есть принудительный переход на `https:`. Это значит:
- домен должен иметь рабочий TLS,
- иначе пользователь увидит ошибку до загрузки приложения,
- особенно это заметно на кастомных доменах, которые ещё не получили сертификат или указывают на неверный хост.

## Практическая диагностика по текущим симптомам

### Если «пропали меню, профиль и бургер»
Сначала проверять:
1. грузится ли Font Awesome CSS с `cdnjs.cloudflare.com`;
2. нет ли ошибок в консоли браузера;
3. завершился ли `bootstrapSession()` и дошёл ли `initApp()`;
4. виден ли `#app` после логина;
5. нет ли на странице блокировщика сетевых ресурсов или CSP/AdBlock‑ограничений.

### Если `taskspyanao.ru` не открывается
Сначала проверять:
1. куда реально указывает DNS домена;
2. привязан ли домен к нужному Vercel‑проекту;
3. выдан ли SSL‑сертификат;
4. открывается ли сайт по `https://` напрямую;
5. не deployed ли проект на платформу, которая не понимает Vercel API routes;
6. не пытается ли хост запускать `npm start` как сервер, хотя тут serverless/SPA.

### Что здесь точно не выглядит правильным
- Один репозиторий содержит сразу **Vercel и Netlify** элементы.
- На фронтенде нет серверной авторизации.
- CORS для API открыт широко.
- Иконки интерфейса полностью завязаны на внешний CDN без fallback.
- `package.json` не соответствует обычному веб‑серверу.

## Итог
С точки зрения кода, меню/профиль/бургер не «пропали» из DOM — они есть. Скорее всего, проблема визуальная или загрузочная: внешний иконочный CDN, неуспешная инициализация JS, либо приложение не доходит до рабочего состояния из‑за инфраструктуры домена.  

С точки зрения подключения сайта, самый вероятный корень — **несовпадение хостинга/доменной записи/SSL с тем, как этот проект реально должен быть развернут**. Этот репозиторий ожидает Vercel‑подобный serverless‑деплой с корректным HTTPS, а не просто запуск `npm start`.
