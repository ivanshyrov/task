# Система Service Desk «Планировщик задач»

Веб-приложение для регистрации, обработки и контроля заявок технической поддержки с синхронизацией данных через SeaTable.

---

## 1. Назначение системы

**Планировщик задач** — это корпоративная система Service Desk, разработанная для автоматизации процессов технической поддержки в организации.

Система позволяет:
- Сотрудникам создавать заявки на техническую поддержку
- Администраторам выполнять заявки и управлять процессом
- Контролировать соблюдение сроков выполнения (SLA)
- Формировать отчёты и статистику
- Управлять пользователями, направлениями и настройками системы

---

## 2. Ролевая модель доступа

Система использует две роли:

| Роль | Описание | Возможности |
|------|----------|-------------|
| **Администратор** | Полный доступ | Управление пользователями, направлениями, всеми заявками, настройки, базы данных, отчёты, создание заявок от имени других администраторов |
| **Сотрудник СП** | Ограниченный доступ | Создание заявок, просмотр своих заявок |

### Первый вход

> **Важно:** Учётные данные по умолчанию УДАЛЕНЫ из соображений безопасности.
> 
> Для первого входа создайте пользователя вручную через SeaTable:
> 1. Откройте таблицу Users в SeaTable
> 2. Добавьте строку с username, password_hash и role
> 3. Пароль сохраняется в SHA-256 хэше

## 3. Аутентификация и авторизация

### 3.1. Как это работает

```
1. Клиент → POST /api/auth с логином/паролем
2. Сервер → Проверяет в SeaTable
3. Сервер → Выдаёт токен (1 час)
4. Клиент → Все запросы с Authorization: Bearer <token>
5. Сервер → Проверяет токен перед каждым запросом
```

### 3.2. Безопасность

- ✅ Токены с TTL 1 час
- ✅ Rate limiting на login (5 попыток, 15 мин lockout)
- ✅ Bearer token в заголовке для всех запросов
- ✅ Только администратор может POST/PUT/DELETE
- ✅ GET задач — без авторизации (публичный просмотр)
- ✅ Idempotency key для предотвращения дубликатов

### 3.3. Клиентский код (пример)

```javascript
// Вход
const res = await fetch('/api/auth', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ login: 'admin', password: 'P@ssw0rd' })
});
const { token, user } = await res.json();

// Запросы с токеном
await fetch('/api/users', {
  headers: { 'Authorization': `Bearer ${token}` }
});

// Выход
fetch('/api/auth', {
  method: 'DELETE',
  headers: { 'Authorization': `Bearer ${token}` }
});
```

## 4. Управление заявками

### 4.1. Функции заявок

| Функция | Описание |
|--------|----------|
| Создание заявки | Форма с полями: направление техподдержки, описание, приоритет, срок, вложения |
| Выбор автора | Администратор может указать автора заявки (из списка администраторов) |
| Редактирование | Изменение полей заявки (в зависимости от статуса и роли) |
| Удаление | Удаление заявок (только администратор) |
| Назначение исполнителя | Назначение ответственного за выполнение (только администратор) |
| Изменение статуса | Переход по статусам |
| Отчёт о выполнении | Текстовое поле с описанием выполненной работы |
| Причина отклонения | Указание причины при отклонении заявки |

### 4.2. Статусы заявок

```
Новая → В работе → Завершена
    ↓              ↓
Отклонена      (завершено)
```

### 4.3. Переходы статусов

| Текущий статус | Доступные переходы |
|---------------|-------------------|
| Новая | В работе, Отклонена |
| В работе | Завершена, Отклонена |
| Завершена | (завершено) |
| Отклонена | (завершено) |

### 4.4. Права на переход статусов

- **Администратор**: может переводить в любой доступный статус
- **Сотрудник СП**: может переводить "Новая" → "В работе", "В работе" → "Завершена" только для своих заявок

## 5. Приоритеты и SLA

### 5.1. Уровни приоритета

| Приоритет | Описание | Срок выполнения (SLA) |
|-----------|---------|----------------------|
| Критический | Срочная проблема, влияющая на работу | 1 день |
| Высокий | Важная проблема | 2 дня |
| Средний | Обычная проблема | 3 дня |
| Низкий | Несрочная проблема | 5 дней |

### 5.2. SLA

- **В SLA**: задача завершена в срок или раньше
- **Нарушен SLA**: задача завершена после истечения срока

## 14. Установка и настройка

### 14.1. Требования

- Node.js 18+
- Аккаунт Vercel
- Аккаунт SeaTable

### 14.2. Развёртывание

1. **Клонирование репозитория**
   ```bash
   git clone <репозиторий>
   cd <папка>
   ```

2. **Установка зависимостей**
   ```bash
   npm install
   ```

3. **Настройка переменных окружения**

   Создать файл `.env.local`:
   ```env
   SEATABLE_API_TOKEN=<ваш токен>
   SEATABLE_BASE_UUID=<uuid базы>
   SEATABLE_SERVER=https://cloud.seatable.io
   SEATABLE_TABLE_NAME=Tasks
   SEATABLE_USERS_TABLE=Users
   SEATABLE_DIRECTIONS_TABLE=Directions
   SEATABLE_ACTIVITY_TABLE=Activity
   ALLOWED_ORIGINS=your-domain.com
   ```

4. **Деплой на Vercel**
   ```bash
   npm install -g vercel
   vercel deploy --prod
   ```

### 14.3. Создание первого пользователя

1. Откройте SeaTable → таблица Users
2. Добавьте строку:
   - username: admin
   - password_hash: (SHA-256 хэш вашего пароля)
   - role: admin
   - full_name: Администратор
3. Теперь можно войти в систему

### 14.4. Настройка SeaTable

1. Создать базу данных
2. Создать таблицы: Tasks, Users, Directions, Activity
3. Настроить колонки согласно документации
4. Получить API токен и UUID базы
5. Настроить переменные окружения в Vercel

## 15. Использование системы

### 15.1. Авторизация

1. Открыть приложение
2. Ввести логин и пароль (из таблицы Users)
3. Нажать «Войти»
4. Получить токен и сохранить в localStorage

### 15.2. Создание заявки (Сотрудник)

1. Нажать «Новая заявка»
2. Заполнить форму (направление, описание, приоритет, срок)
3. Нажать «Создать»

### 15.3. Работа с заявками (Администратор)

1. Просмотр: клик по строке
2. Редактирование: изменить статус, исполнителя, приоритет
3. Фильтрация: использовать панель фильтров

### 15.4. Отчёты

1. Перейти в раздел «Отчёты»
2. Выбрать базу данных
3. При необходимости выбрать исполнителя (для администратора)
4. Указать период (дата с / по)
5. Нажать «Применить»
6. Нажать «Сбросить» для сброса фильтров
